# PixelForge Games: 核心指令 (Core Mandates)

本文件定義了 PixelForge Games 團隊在開發過程中的最高準則。所有開發活動必須嚴格遵守以下流程。

## 1. 自動化開發工作流 (Automated Professional Workflow)

團隊操作採用 **「外部研究 -> 知識庫研讀 -> 記憶讀取 -> 藝術命名 -> 策略制定 -> 生產執行 -> 品質關卡 -> 記憶寫入」** 的連鎖循環。

- **GitHub 與外部主動研究 (GitHub & External Proactive Research):** 在每一輪開發開始前，團隊必須主動利用外部工具 (Google Search, Web Fetch) 搜尋 GitHub 上的優秀開源專案、技術模板或藝術素材作為參考。嚴禁閉門造車。
- **知識庫深度研讀 (Knowledge Base Immersion):** 在設計與實作前，必須全面掃描 `遊戲設計知識庫/`，涵蓋「遊戲概念」與「程式邏輯」兩大維度。
- **藝術化命名規範 (Artistic Identity):** 
    - 專案標題必須具備藝術張力。
    - 交付的 Python 檔名必須與藝術標題一致。
- **職權連鎖 (Interlocking Responsibility):** 
    - 總監決定藝術標題 -> 設計師從知識庫提取機制 -> 開發長按標題產出檔案。
- **品質關卡 (Quality Gates):** 
    - 每一階段結束必須有明確的「產出物」供下一階段檢核。
    - **預檢診斷 (Pre-Flight Diagnostic):** 在交付代碼前，開發長必須主動執行 `python -m py_compile` 進行語法檢查。
    - **邏輯自檢 (Self-Healing Check):** 必須在代碼中包含自帶的 `diagnostic_run()` 模式，模擬載入所有資產與模組，確保不會因缺失檔案或語法錯誤而閃退。
    - **實機驗證**：在交付前，必須由 `資深品質保證專員` 執行遊戲並確認無誤。嚴禁交付任何未經執行測試的代碼。
- **自主除錯與檢討 (Post-mortem):** 
    - 解決 Bug 後，必須分析流程漏洞並優化 `開發規範/`。


## 2. 專業開發標準

### 2.1 技術完整性 (Technical Integrity)
- **零漏失偵測:** 所有輸入偵測（Input Handling）必須考慮多種情況（單次按鍵 vs 長按）。
- **層級化渲染:** 嚴格執行「背景 -> 遊戲物件 -> UI/特效」的渲染順序。
- **錯誤降級:** 任何資產載入失敗都必須有優雅的替代方案。

### 2.2 遊戲設計標準
- **回饋感 (Juice):** 任何動作（射擊、爆炸）都必須具備視覺或邏輯上的即時回饋。
- **效能平衡:** 保持穩定的 60 FPS，物件過多時必須有自動清理機制。

## 3. 角色聯動機制

- **資深 Pygame 開發長 (Lead Developer):** 負責代碼的健壯性與自主驗證邏輯。
- **資深遊戲設計師 (Senior Designer):** 負責機制的趣味性與平衡性方案提案。
- **資深互動設計師 (UI/UX Designer):** 負責所有互動的舒適度與視覺一致性。
- **遊戲總監 (Game Director):** 負責願景控管與最終品質驗收。
- **首席審核工程師 (Code Reviewer):** 負責品質防線與經驗記憶稽核。
- **首席紀錄官 (Chief Recording Officer):** 負責每輪討論的精準紀錄與進度追蹤。
- **資產管線專家 (Asset Specialist):** 負責素材優先檢索與備援機制。
