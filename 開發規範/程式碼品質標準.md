# Guideline: 專業開發與驗證流程 (Professional Development & Validation)

## 1. 開發前置檢查 (Pre-development Analysis)
在撰寫任何代碼前，必須完成以下分析：
- **邏輯依賴**: 檢查新功能與現有系統（如精靈組管理）的交互。
- **潛在衝突**: 預判按鍵偵測、碰撞邏輯或渲染順序可能出現的 Bug。

## 2. 代碼品質規範 (Code Quality)
- **命名規範**: 使用具備語義化的命名（例如 `cooldown_timer` 優於 `t1`）。
- **註解標準**: 每一個關鍵邏輯區塊（類別、函式、複雜循環）必須有詳細的專業註解。
- **解耦設計**: 盡量減少全域變數的使用，通過對象屬性傳遞狀態。

## 3. 引用完整性 (Reference Integrity)
- **配置防禦化 (Defensive Config)**: 禁止使用字典儲存核心配置。必須使用靜態類別（Static Classes）管理顏色、路徑與字型，以利於靜態檢查。
- **類型穩定性**: 確保變數在整個生命週期中保持一致的類型。

## 4. 自主驗證協議 (Autonomous Validation Protocol)
這是避免 Bug 交付給使用者的關鍵步驟：
- **邊界測試**: 測試螢幕邊緣移動、極高速射擊、大量敵人生成時的表現。
- **邏輯乾跑**: 手動模擬事件迴圈，檢查按鍵標記位元（Flags）是否能正確重置。
- **視覺一致性**: 檢查 UI 是否在所有狀態（開始、遊戲中、結束）都能正確對齊與顯示。

## 4. 故障排除流程 (Troubleshooting)
若使用者發現 Bug，開發團隊必須：
1. 立即停止當前開發，進行問題重現。
2. 分析為何先前的「自主驗證」未能捕捉此問題。
3. 修復 Bug 並同時優化驗證邏輯，防止類似問題再次發生。
